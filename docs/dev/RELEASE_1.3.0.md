# RELEASE_1.3.0.md ‚Äî CentaurTechnicalIndicators-JS

Release plan for `1.3.0`, structured as one Claude Code session (one PR) per
unit of work. Baseline is **`main` (1.2.2)**: `.expect()` panics present,
`standardIndicators` / `ti-engine` / underscore-README rot present, no clippy
gate in CI.

The release is bookended: **PR 1** bumps the Rust dependency only; the **final
PR** cuts the package version and finalizes the changelog. Everything else lands
in between.

All test vectors below are verified against Rust 1.3.0 by executing both crate
versions. Cited file/line numbers reflect `main` at plan time ‚Äî re-confirm them
at session start, since intervening PRs shift line numbers.

---

## Standing rules (apply to every PR)

- **Governing docs are ground truth.** Do not edit this file from within a CC
  session; treat it as read-only instruction. Surface drift; do not silently
  resolve it.
- **Stop and report on any blocker.** Do not work around an unexpected obstacle.
  Stop, report, wait.
- **Scope discipline.** Touch only the files named in the PR. Note unrelated
  issues separately; do not bundle.
- **Validation gates (all must pass before PR):**
  `cargo fmt --check` ¬∑ `cargo clippy` ¬∑ `npm run build` ¬∑ `node --test`
  - **Clippy strictness:** plain `cargo clippy` (warnings permitted) for every
    PR before PR 8 (PRs 1‚Äì7 and 6b). **PR 8 introduces `-D warnings`; from PR 8
    onward the strict gate applies.** Earlier PRs must not be held to
    `-D warnings` ‚Äî the pre-existing chaikin-arg lint is only fixed in PR 8.
- **Changelog coupling.** `[Unreleased]` starts empty on `main`. Every PR
  appends its own entry under `## [Unreleased]`. The final PR renames
  `[Unreleased]` to `[1.3.0]` and reopens an empty `[Unreleased]`. No
  user-facing PR merges with an empty changelog entry.
- **PR report format:** `Summary` ¬∑ `Scope` ¬∑ `Compatibility` ¬∑ `Validation` ¬∑
  `Changelog` (per `AGENTS.md`; `ai-policy.yaml` is removed in PR 6b).

## PR 2 ‚Äî Error-handling conversion: `chart_trends` (creates shared helper)

**Depends on:** PR 1. First error-handling PR; runs before PR 3.

**Objective:** introduce the shared error adapter and convert the `chart_trends`
wrappers from `.expect(...)` panics to shaped JS errors.

**Changes:**
- Create `src/jsutil.rs` with the adapter
  `pub fn js_err(err: impl std::fmt::Display) -> JsValue {
  JsValue::from_str(&err.to_string()) }`; register `pub mod jsutil;` in
  `src/lib.rs`. **Name it `js_err`** (not `js_error`).
- In `src/chart_trends.rs`, convert `.expect(...)` call sites to return
  `Result<T, JsValue>` via `.map_err(js_err)` ‚Äî both scalar (`f64`) and `Array`
  (tuple/series) returns.

**Tests:** add `assert.throws(...)` cases for representative invalid inputs to
`test/chartTrends.node.test.js`.

**Compatibility:** failure *mode* changes (panic ‚Üí thrown `Error`); success
behavior unchanged. Non-breaking (already threw at call time); stays a minor.
**Why this is a real fix:** wasm32 defaults to `panic=abort` and the crate sets
no panic hook, so a Rust panic traps the singleton wasm instance and poisons
subsequent calls into the Node/bundler module; `Result` throws cleanly and keeps
the instance usable.

**Out of scope:** other modules' wrappers (PRs 5a‚Äì5h); no new bindings.

## PR 3 ‚Äî Favorable-move bindings

**Depends on:** PR 1 and **PR 2** (the two functions are fallible ‚Äî see below ‚Äî
so they wrap through PR 2's `js_err`).

**Objective:** mirror the two new Rust 1.3.0 `chart_trends` functions into JS.

**Signatures (verified):** both are
`pub fn ‚Ä¶(prices: &[f64], index: usize, period: usize) -> crate::Result<f64>`
‚Äî **fallible**, new in 1.3.0.

**Changes:**
- `src/chart_trends.rs`: add `chart_trends_peakFavorableMove` and
  `chart_trends_valleyFavorableMove`, returning `Result<f64, JsValue>` via
  `.map_err(js_err)`.
- Export `peakFavorableMove` / `valleyFavorableMove` under `chartTrends` in
  **all three** entry points (`index.js`, `index.node.js`, `index.web.js`).
- `index.d.ts`: add both to the `ChartTrends` interface with JSDoc and the
  `@see` playground-URL pattern matching existing entries.

**JSDoc semantics (verified accurate):** inclusive window `[index+1,
index+period]`. `peakFavorableMove` = `prices[index] ‚àí min(window)` (largest
downward move; **not floored** ‚Äî negative if no drop). `valleyFavorableMove` =
`max(window) ‚àí prices[index]` (largest upward move; not floored). Throws on:
empty data, `period == 0`, or `index + period >= len`.

**Tests (verified vectors):**
```
peakFavorableMove([107,104,100,102], 0, 3)        -> 7.0
valleyFavorableMove([100,102,107,104,100], 0, 3)  -> 7.0
peakFavorableMove([100,101,102,103], 0, 3)        -> -1.0   (no drop; not floored)
valleyFavorableMove([105,104,103,102], 0, 3)      -> -1.0   (no rise; not floored)
valleyFavorableMove([10,1,1,20,99], 0, 3)         -> 10.0   (idx3 captured, idx4 excluded)
peakFavorableMove([50,99,99,30,1], 0, 3)          -> 20.0   (idx3 captured)
peakFavorableMove([100,101,102], 1, 3)            -> THROWS  (window past end)
{peak,valley}FavorableMove(prices, 0, 0)          -> THROWS  (period 0)
{peak,valley}FavorableMove([], 0, 3)              -> THROWS  (empty)
```

**Changelog (under `[Unreleased]`):** `### Added ‚Äî JS/WASM bindings for
chart-trend favorable-move functions (peakFavorableMove, valleyFavorableMove).`

# guard against a 1.2.2 regression ‚Äî keep only as 1.3.0 behavior confirmation):
chartTrends.peaks([110,109,108,120], 2, 2)    -> [[110,0],[120,3]]
chartTrends.valleys([90,91,92,80], 2, 2)      -> [[90,0],[80,3]]

# all-NaN peaks/valleys -> EMPTY ARRAY, not a throw:
chartTrends.peaks([NaN,NaN,NaN,NaN], 2, 1)    -> []   (assert.deepEqual [], NOT assert.throws)
chartTrends.valleys([NaN,NaN,NaN,NaN], 2, 1)  -> []

# all-NaN single fns -> NaN (no period arg):
trendIndicators.single.aroonUp([NaN,NaN,NaN])                 -> NaN  (assert Number.isNaN)
trendIndicators.single.aroonDown([NaN,NaN,NaN])               -> NaN
momentumIndicators.single.stochasticOscillator([NaN,NaN,NaN]) -> NaN

## PR 7 ‚Äî CommonJS export path

**Depends on:** nothing.

**Problem (corrected diagnosis):** `package.json` has `"type": "module"` and
maps `require` ‚Üí `index.node.js`. That file is **well-formed** ESM (uses
`createRequire(import.meta.url)`) ‚Äî it is not malformed. The defect is that
`require()` of an ESM entry throws `ERR_REQUIRE_ESM` on the Node 20 floor
(`require(esm)` landed in 22.12) and, on Node 24, returns a namespace with keys
but `undefined` values. A CJS consumer breaks either way.

**Fix:**
- Add `index.node.cjs`: CommonJS requiring the `dist/node` glue and re-exporting
  the namespaces.
- `exports.require` ‚Üí `./index.node.cjs`; add `index.node.cjs` to `package.json`
  `files`.
- Add `test/require.cjs` smoke test.

(The `docs/REPO_MAP.md` mislabel of `index.node.js` is not fixed here ‚Äî that file
is deleted in PR 6b.)

**Out of scope:** no change to the ESM entry points or the namespace API.

## Deferred ‚Äî configurable / parameterized indicator variants (Codex item 6)

**Status:** out of 1.3.0. Gated on an unresolved decision; do not implement
until that decision is "yes."

**Gating decision (answer first):** should the public JS API expose
configurable/parameterized indicator variants at all, or does
optimizable-parameter flexibility live exclusively in `crt-research-engine`? The
JS library is a faithful WASM mirror of the Rust public surface for retail use;
parameter optimization is the research engine's job. Exposing parameterized
variants in JS creates a second, hand-maintained configurability path the Rust
enum API doesn't have. That may be worth it, but it's a positioning decision,
not a binding task. If the answer is "research engine only," this entry becomes
*never* and should be deleted.

**Background:** Rust exposes parameterized enum variants JS currently and
deliberately omits ‚Äî `MovingAverageType::Personalised { alpha_num, alpha_den }`,
`ConstantModelType::PersonalisedMovingAverage { alpha_num, alpha_den }`,
`DeviationModel::StudentT { df }`,
`DeviationModel::EmpiricalQuantileRange { low, high, precision }`,
`DeviationModel::CustomAbsoluteDeviation { config }`. `index.d.ts` documents the
`Personalised` omission as intentional ‚Äî a considered exclusion, not an
oversight.

**Why it isn't a simple binding:** wasm-bindgen numeric enums can't carry struct
payloads, so these can't route through the existing enum API. Exposure needs
net-new function or config-object surface, maintained in lockstep across the
Rust wrapper, all three entry points, and `index.d.ts`, indefinitely. Every
variant exposed is a permanent cross-language obligation.

**Proposed approach (only if approved):** dedicated functions, not overloaded
enums; keep the existing enum API stable. Sketch ‚Äî
`movingAverage.single.personalised(prices, alphaNum, alphaDen)` and `.bulk`
(+`period`); `studentTAdjustedStd(prices, df)`; `empiricalQuantileRange(prices,
low, high, precision)`; explicit band variants
`candleIndicators.single.movingConstantBandsStudentT(...)` /
`...movingConstantBandsEmpiricalQuantile(...)`.

**Open sub-decisions (resolve at scoping):** namespace placement (the natural
home `basicIndicators` doesn't exist in JS ‚Äî a separate upstream-parity gap);
naming for variant-specific band functions; whether to expose all variants or
only the subset the retail surface / optimizer consumes (default: smallest set
that does real work).
